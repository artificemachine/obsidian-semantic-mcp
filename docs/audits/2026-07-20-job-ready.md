**Second scope limit:** this machine already has Ollama running, `nomic-embed-text` pulled, and `obsidian-semantic` registered with Claude Desktop and OpenCode. A true stranger's machine would have none of it, so the "clean environment" simulation is partial — the prerequisites the wizard reported as ✓ are exactly the ones a new user would have to install.

## Correction — 2026-07-20 (post-audit)

**Stage 5 finding "starlette/uvicorn are pinned but never imported — pure CVE surface with no functional role" was wrong in its conclusion.**

What the audit measured was correct: `grep -rn 'starlette\|uvicorn' src/` returns zero hits, and the dashboard is stdlib `http.server`. What the audit *inferred* from that was not. Both packages are unconditional requirements of the `mcp` SDK, which this project genuinely uses:

```
$ python3 -c "import importlib.metadata as md; print(md.requires('mcp'))"
  starlette>=0.27
  uvicorn>=0.31.1; sys_platform != 'emscripten'
  httpx>=0.27.1
  pyjwt[crypto]>=2.10.1
```

Consequences:

1. **Removing the explicit pins does not remove the 4 starlette CVEs.** They arrive transitively via `mcp` regardless. Verified: `import starlette` still succeeds after the pins were dropped. The Stage 4 CVE count is unaffected by this change.
2. Dropping the pins is still correct, for a different and weaker reason: the project should not claim ownership of a version constraint on a package it never imports and does not audit. That is hygiene, not remediation.
3. `httpx` and `pyjwt` are in the same category — declared as direct dependencies, never imported under `src/`. The audit did not measure them and they remain out of scope.

**The real fix for those CVEs is bumping `mcp` 1.26.0 → 1.28.1**, which is already Stage 4's top recommendation and independently closes CVE-2026-52869/52870/59950.

Recorded here rather than by editing the Stage 5 section, so the original claim and its correction both stay visible.
